## 2. Insert Data

### 2.1 Insert Matched Variables for Existing Stations
- Inserted all variables where both station name and variable name could be matched.
- Only new data (not already in the database) was added, determined by comparing the earliest and latest times of the raw data and database records.
- **Process code:** `1.match_lat_lon_rows_only_new.ipynb`
- **Summary file:** `./rows_output/Fern_raw_db_station_matched_v1.csv`
- **Progress:** 251 of 433 variables processed across 35 stations.



### 2.2 Insert Variables for New Stations

- Prepared rows for 6 stations not yet in the database: `Atlin School`, `CrookedLk`, `IBB2Wx`, `IBB3Wx`, `Inklin`, `SeebachWx`.
- Variables from these stations that match existing database variables have been inserted.
- **Process code:** `2.match2_insert.ipynb`
- **Summary file:** `Fern_matched_v2_raw_db_station.csv`
- **Progress:** 58 of 182 variables ready.


### 2.3 Manual Variable Matching

- 97 variables remain due to ambiguous or unmatched names.

  - `Water_Content_m3_m3_5_cm`
  - `Water_Content_m3_m3_15_cm`
  - `Water_Content_m3_m3_30_cm`
- `Soil_Temp` and `Snow_Depth` for 8 stations have also been inserted.
- **Progress:** 40 of 97 variables processed.
- **Process code:** `3.var_match_insert.ipynb`
- **Summary file:** `Fern_manually_matched_v3_raw_db_station.csv`

## 3. Double Checking - The inconsistence of native_id of same station

- I compared the raw data with the updated database by checking the earliest and latest timestamps.
  
- All rows that have no missing values should have been inserted correctly.
  
- The issue appears to come from native_id. For some stations, the native_id in the raw file is different from the native_id stored in the database. Because of this mismatch, the script treated those records as if they were new stations, and inserted them again.

- Since the rows code didn't give "station_name", the inserted “new” rows only had native_id. I updated those missing station names by setting them equal to their native_id. After that, I saw that the same station now shows up with two different native_ids... 

- Then we need to decide how to handle these conflicting IDs. It’s possible that the station IDs were changed at some point? And also compare the latitude, longitude, and elevation; then we can decide if we want to merge the native_ID.

- **Process code:** `5.double_check.ipynb`
- **Summary file:** `5.Fern_db_updated_summary.csv`, `5.double_check_Fern_raw_db_matches.csv`


## 4. Summary, Reflections, and Recommendations


  This task provided a good chance to work with raw observation data and the database. I gained a better understanding of the database structure, raw data format, SQL usage in Python, and developed tools for PCIC’s database workflow.  
  
  Experimenting with different coding approaches and summary formats helped me discover the most efficient ways to sort, compare data, and record results. These skills will be valuable for future database-related tasks.

- **Challenges:**  
  The most difficult parts were sorting and comparing raw data with the existing database, and handling unmatched station names and variable names. Developing strategies to efficiently manage and document these cases was important.

  It’s also important to understand any possible updates or changes to the stations themselves. For example, the inconsistent native_id might be due to a station update or a transfer in the source system, which means we may need to communicate with the data provider. At the same time, this gives us a good opportunity to better understand the observation station metadata.

- **Code automates:**  
  All summary tables were generated programmatically, none of manualy handle. Only a few manully checks were made to handle the station and variable match, and these were reflected in the codes.
