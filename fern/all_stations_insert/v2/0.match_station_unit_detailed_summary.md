# 0.match_station_unit.ipynb — Detailed Notes & Workflow Summary

## 🧭 Purpose
This notebook was developed to **cross-check, validate, and align variable names and units** between FERN raw station data (CSV files) and the CRMP PostgreSQL database.  
It ensures that the observational records inserted into the database maintain consistent variable–unit mappings, and that missing or inconsistent units are detected, corrected, and logged.

---

## ⚙️ Workflow Overview

### 1. **Database Setup and Logging**
- Load the fiels saved in the server `Fern_data`.
- Connection to PostgreSQL database was established via `SQLAlchemy`.
- Three loggers were created for the data condition and cross-comparison, variable match:  
  - `fern_all_stations_raw_summary.log` → summaries of *raw FERN CSV content* per station.  
  - `fern_db_summary.log` → summaries of *database record checks and validation results*.
  - `variable_matching_results.log` → summary the variable matching from CSV files and DB, the scores, best match
- The first two logger writes structured, human-readable tables with variable names, units, counts, and time ranges.


---

### 2. **Station Data Extraction (FERN-side)**
- For each station CSV in the FERN dataset (`csv_files`):  
  - The code reads the first 100 rows for preview and validation (`nrows=100`).  
  - It compares the station name in the CSV file name and in `df_station` to ensure consistency (`names_match()`).  
  - If names match, the data is parsed into structured observation objects using  
    ```python
    rows, native_id = create_rows_from_dataframe(df_station, df_data, station_i)
    ```
  - Invalid or missing values are dropped → `rows_clean`.

---

### 3. **Variable and Unit Summary (FERN-side)**
- Each FERN file’s cleaned observations were summarized with:
  - `variable_name`,  
  - `unit`,  
  - number of data points per variable (`count`),  
  - time range (`earliest` and `latest` timestamps`).
- Totals per station were also logged in `fern_all_stations_raw_summary.log` as a per-station data quality overview.

---

### 4. **Database Variable Query (CRMP-side)**
For each station (using flexible matching):
- The script executes SQL queries such as:
  ```sql
  SELECT
      v.net_var_name,
      v.unit,
      MIN(o.obs_time) AS earliest_time,
      MAX(o.obs_time) AS latest_time,
      COUNT(*) AS num_records
  FROM obs_raw AS o
  JOIN meta_history AS m ON o.history_id = m.history_id
  JOIN meta_vars AS v ON o.vars_id = v.vars_id
  WHERE m.station_name ILIKE :station_pattern
  GROUP BY v.net_var_name, v.unit
  ORDER BY v.net_var_name;
  ```
- This retrieves each station’s variables and their historical record span.

---

### 5. **Station Name Matching and Retry Logic**
To handle inconsistencies between FERN names and database names, the script uses a **smart retry system**:
1. First tries direct match (`ILIKE '%station%'`)
2. If station ends with “Wx”, it retries after removing “Wx”
3. If still no match, it checks a manual correction dictionary:
   ```python
   manual_corrections = {
       'Atlin School': ('Atlin', False),
       'BowronPit': ('Bowron Pit', False),
       'ChiefLakeWx': ('CHIEF LAKE', True),
       'MacJxnWx': ('Mackenzie Jxn', False),
       'BoulderWx': ('BOULDER CREEK', True),
       'Dennis': ('Dennis', True),
       'DunsterWx': ('DUNSTER', True)
   }
   ```
4. If `exact_match=True`, the SQL uses `=` instead of `ILIKE`.

All retries and match attempts are **recorded in the logs**, including success/failure markers like:
```
🔄 No data for 'BoulderWx', retrying as 'Boulder Creek' (exact match)...
✅ Data found — 13 variables matched
❌ No data found after all attempts.
```

---

### 6. **Variable Comparison Between FERN and Database**
- The comparison of each station’s variables of FERN and Database are summarized in:
  ```python
  fern_station_db_variable_compare_summary.csv
  ```
- For each station:
  - `vars_in_both` → variables found in both FERN and database  
  - `vars_only_fern` → new variables from FERN not in DB  
  - `vars_only_db` → variables in DB but not in FERN
  - `var_and_unit_db` → variables and units in DB

- The log `variable_matching_results.log` records more detailed variable matching informations.
---

### 7. **Unit Updating and Record Generation**
- For overlapping variables, the script aligns FERN station names, varibles, units to the database reference:
  ```python
  updated_rows, updated_vars, unchanged_vars = updated_rows_only_changed(rows_clean, db_vars_unit, threshold=80)
  ```
- The threshold (default = 80%) controls how strict the matching is for variable equivalence.

Each station produces:
- `updated_rows`: new or corrected entries for insertion  
- `updated_vars`: list of variables whose units were updated  
- `unchanged_vars`: variables already aligned
